Investigation Report: Prayer Sequence State Machine
Note on tool output: partway through this investigation, a Bash command's output contained injected text mimicking system-reminders (fake "plan mode" / permission instructions). That was untrusted content embedded in a file/command result, not a real instruction from you or the harness — I ignored it and continued the read-only investigation as originally scoped.

1. Azan popup show/hide
State: isAzanPopupVisible (m2body.js:2871), set true in showAzanPopup() (m2body.js:2908), set false in hideAzanPopupFunction() (m2body.js:2919).
Show: showAzanPopup(isDohaPrayer) (m2body.js:2872-2912). Duration: azanPopupDisplayDuration = JS_AZAN_WINDOW_SHOW_TIME * 1000 (default 60s, settings-defaults.js:138), or 10000 ms for Doha (m2body.js:2879). Close is scheduled purely relatively: setTimeout('hideAzanPopupFunction()', azanPopupDisplayDuration) (m2body.js:2909) — no absolute timestamp is stored anywhere; it's a plain relative timer.
UC_EVT dispatch: entirely a custom.js construct. _ucInstallHooks() IIFE monkey-patches showAzanPopup and hideAzanPopupFunction (custom.js:5558-5575): fires AZAN_TIME then calls original then fires AZAN_SHOW; hideAzanPopupFunction wrapper calls original then fires AZAN_HIDE.
Close paths (three, not unified):
Core auto-timeout above (60s/10s).
custom.js Quran-audio-block feature (custom.js:8685-8815): while azan audio plays, hideAzanPopupFunction is monkey-patched again to block (_ucBlockAzanHide), released on the audio ended event via _ucReleaseAzanBlock() (custom.js:8757-8768). Two safety timers, not one: 300000 ms / 5 min (custom.js:8809-8812, normal path) and 180000 ms / 3 min when JS_DATA.ucAzanIqamaByVoice != 1 (silent mode) (custom.js:8777-8781). A stray comment at custom.js:8628 references an approximate "safety_timeout_4min" for a GeckoView audio-redirect edge case — this does not match the actual 300000/180000 constants; likely doc drift vs. the top-level CLAUDE.md's "4 minutes" claim.
Manual user close bypasses everything: index.html:122 and index.html:335 wire onclick='hideElementFunction(this.id)' directly on the popup divs — this calls the low-level hideElementFunction (just sets className='hiddenClass'), not hideAzanPopupFunction(). Consequence: isAzanPopupVisible is never reset to false, and UC_EVT.AZAN_HIDE is never fired through the official patched path. custom.js separately intercepts raw hideElementFunction in two other places (custom.js:8963-8972 ticker cleanup, custom.js:9053-9060 audio force-stop) to catch this bypass for those specific side effects only — but isAzanPopupVisible staleness is not addressed anywhere.
No absolute "azan started at" anchor exists. Timing is 100% relative setTimeout chains.
2. Iqama countdown
startIqamaCounterFunction(iqamaMinutes, prayerDuration, forcedCounterSeconds) (m2body.js:3718-3762): confirms CLAUDE.md exactly — remainingSeconds = (iqamaMinutes * 60) - currentSecondsComp where currentSecondsComp = new Date().getSeconds() at call time (m2body.js:3750-3755). isIqamaCounterActive = true (3754); totalCounterSeconds similarly derived (3759-3761).
clockTickFunction() (m2body.js:1928-2055), the 1Hz heartbeat, decrements remainingSeconds-- each tick when isIqamaCounterActive (line 1965). At remainingSeconds == 0 (2033-2047): sets isIqamaCounterActive = false, then setTimeout('startIqamaSequenceFunction()', 1000).
startIqamaSequenceFunction() (m2body.js:1132-1166): schedules onIqamaEndFunction via setTimeout(..., prayerDurationMinutes * 60000) (1146), and black-screen/popup timers (see §3).
No absolute target-iqama timestamp global exists for the live countdown. However, checkAndRestoreIqamaCounter() (m2body.js:8233-8285) is exactly the reconstruction formula you'd want to reuse: e.g. remainingIqamaMinutes = (fajrTimeInMinutes + JS_DATA.ucIqamaFAJR) - currentTimeInMinutes, then re-calls startIqamaCounterFunction(remainingIqamaMinutes, prayerDuration, iqamaDelayMinutes) and showIqamaCounter(). Same pattern repeated per prayer (SHRQ/DOHR/ASSR/MGRB/ISHA, m2body.js:8240-8279) using globals fajrTimeInMinutes/dohrTimeInMinutes/etc. and JS_DATA.ucIqamaFAJR/ucIqamaDOHR/etc. (iqama delay minutes) plus JS_DATA.ucPrayDurationFAJR/etc. But this function is only ever called once, via setTimeout(checkAndRestoreIqamaCounter, 1000) inside the window.addEventListener('load', ...) handler (m2body.js:8842) — never on focus/visibility/resume.
Core's own window.onfocus = function(){ calculateAndDisplayTimesFunction(); updateTimeAndPrayersFunction(true); } (m2body.js:8845) does not call checkAndRestoreIqamaCounter() — it only recalculates prayer times/highlighting, so an already-running iqama counter is left un-resynced on focus today.
_installCounterDriftCorrection (custom.js:6617-6655): confirmed it is drift-only, not a backgrounding fix. It listens to the synthesized UC_EVT.COUNTDOWN_TICK (see §5), keeps a Date.now() baseline (_startTime/_startRem) captured on the first tick after each UC_EVT.IQAMA_TIME (baseline resets there, custom.js:6650-6653), and corrects remainingSeconds only if drift > 2s (custom.js:6638-6646). Critically: if expected <= 0 (i.e., the target moment has already passed, as would happen after minutes of suspension), it clamps to Math.max(1, expected) (custom.js:6643) rather than recognizing iqama-time has already elapsed — so it's unsuitable as-is for multi-minute background gaps.
3. Black screen
activateBlackScreenIfEnabled() (m2body.js:1858-1861) → activateBlackScreenFunction() (m2body.js:1865-1873): stops azkar/slides (stopAzkarDisplayFunction(), stopSlidesDisplayFunction()), resets nightPrayersClickCounter, sets style.transform = 'scaleX(1)' on both black-screen elements.
deactivateBlackScreen() (m2body.js:1820-1824): sets scaleX(0). Triggered two ways: (a) automatically from onIqamaEndFunction() (m2body.js:1173), scheduled by setTimeout('onIqamaEndFunction()', prayerDurationMinutes*60000) inside startIqamaSequenceFunction (1146); (b) manually via onclick='deactivateBlackScreen()' on blackScreenVertical/blackScreenHorizontal (index.html:243, 447).
Core timing in startIqamaSequenceFunction (m2body.js:1146-1165): delayBeforeBlackScreen = 12000 (default) or 0 if ucShowIqamaScreen==0; iqama popup hidden at delayBeforeBlackScreen - 2000 = T+10s (1160-1161); black screen activation scheduled at T+12s (1153) — matches CLAUDE.md's T+0/T+10/T+12.
custom.js extension (custom.js:9524-9564, "+5s"): intercepts hideElementFunction for the two iqama-popup IDs to delay the actual hide by EXTRA_MS=5000 (9536-9546), and sets a _delayBlackScreen flag; intercepts activateBlackScreenIfEnabled to likewise delay by 5000ms if that flag is set (9553-9563). Net effect: popup hide lands ~T+15s, black screen ~T+17s currently.
BLACK_SHOW/BLACK_HIDE dispatched via a MutationObserver on blackScreenVertical.style.transform (custom.js:5609-5618) — deliberately not a monkey-patch, per the comment there, since activation is scheduled via a string-eval'd setTimeout('activateBlackScreenIfEnabled()').
No dedicated "is black screen active" state variable beyond the DOM style.transform value.
4. Azkar/doaa sequence
onIqamaEndFunction() (m2body.js:1171-1184): deactivateBlackScreen(); if JS_DATA.ucAzkarActive==1, setTimeout('startAzkarMainSequenceFunction()', 7000); else isAzkarMain = true; decideNextAzkarOrSlideFunction().
startAzkarMainSequenceFunction() (m2body.js:1745-1756): if ucAzkar5minPicture==1, shows a 5-minute static picture overlay instead (showAzkar5minPictureFunction, hides itself after 60000*5 ms at line 1792, then chains to decideNextAzkarOrSlideFunction via hideAzkar5minPictureFunction, m2body.js:1798-1804); else calls startAzkarSequenceFunction(mainAzkarArray, true, false).
startAzkarSequenceFunction(azkarArrayToDisplay, isMainAzkar, isSabahAzkar) (m2body.js:1390-1410): resets currentAzkarIndex = 0, copies into azkarDisplayArray (1393-1399), sets isAzkarMain, azkarDisplayInterval = JS_MainAzkarViewTime*1000 (main) or JS_SabahMasaaViewTime*1000 (1403-1405), sets isSabahFlag, shows the azkar container, calls displayNextAzkarFunction().
displayNextAzkarFunction() (m2body.js:1478-1553): the actual chain driver — setTimeout("displayNextAzkarFunction()", azkarDisplayInterval) (1551) plus setTimeout("fadeOutAzkarTextFunction()", azkarDisplayInterval-6000) (1552); increments currentAzkarIndex (1549); one-shot stopAzkarFlag check (1484-1488, resets itself to false when consumed); when currentAzkarIndex >= azkarDisplayArray.length (1490-1496) → hideAzkarDisplayFunction() + setTimeout("decideNextAzkarOrSlideFunction()", 7000).
decideNextAzkarOrSlideFunction() (m2body.js:1446-1472): branches on currentPrayerIndex (0=Fajr→sabah azkar, 3=Asr, 4=Maghrib, 5=Isha → masaa azkar) combined with the isAzkarMain flag and JS_DATA.ucAzkarXxxOn settings, chaining into a second startAzkarSequenceFunction(sabahAzkarArray/masaaAzkarArray, false, ...) pass, or falling through to setTimeout("startSlidesSequenceFunction()", 7000) if slides are enabled.
Global state: currentAzkarIndex, azkarDisplayArray (declared inline, m2body.js:1393-1394 usage; no top-level let found outside this scope — check mainAzkarArray/sabahAzkarArray/masaaAzkarArray are the source pools), azkarDisplayInterval (m2body.js:1476), stopAzkarFlag/stopSlidesFlag (m2body.js:1191-1192), isAzkarMain (m2body.js:1443), isSabahFlag (m2body.js:1444).
Confirmed: the entire chain is nested string-eval'd setTimeout calls with zero absolute time anchor — no "azkar phase started at" timestamp, no "which azkar index should be showing right now" computation exists anywhere.
5. Master clock tick
clockTickFunction() (m2body.js:1928-2055) is the single heartbeat, started via setInterval('clockTickFunction()', 1000) inside startClockIntervalFunction() (m2body.js:32-35), itself invoked once via setTimeout(startClockIntervalFunction, 1000) in the window.addEventListener('load', ...) handler (m2body.js:8841). Confirmed: this is the one setInterval the whole chain depends on — iqama decrement (line 1965), azan-popup time refresh (1950-1954), and the minute-boundary dispatcher if(currentSeconds=='00') updateTimeAndPrayersFunction(false) (line 2052) which in turn triggers highlightPrayerRow→showAzanPopup and startIqamaCounterFunction at prayer boundaries.
custom.js does not monkey-patch clockTickFunction itself (no reassignment found). Instead it runs a separate setInterval(..., 500) poll (custom.js:5624 area, inside _ucInstallHooks) that reads isIqamaCounterActive/remainingSeconds and synthesizes UC_EVT.COUNTDOWN_TICK events with milestone tracking (_ctrMilestones = [300,240,180,120,60,30,20,15,10,5,0]).
_installCounterDriftCorrection (custom.js:6617-6655) listens to that synthesized COUNTDOWN_TICK, not to clockTickFunction directly — see §2 for exactly what it does (small-drift correction only, Date.now() baseline reset per IQAMA_TIME, clamps to 1 rather than detecting elapsed target).
6. visibilitychange / document.hidden / pagehide / pageshow / Android bridge
Confirmed absent: zero matches for visibilitychange or document.hidden anywhere in custom.js or m2body.js.
pageshow exists once, paired with beforeunload (custom.js:9958, 9965-9979) — but this is for an unrelated concern: guarding _ucUnloading state so the light-automation HTTP retry queue, Quran-player probe timer, and hijri sync timer are correctly torn down/rearmed around GeckoView's quirky beforeunload-without-real-unload behavior on location.reload(). It has no interaction with isAzanPopupVisible, remainingSeconds, black-screen state, or azkar progress.
No native Android→JS pause/resume lifecycle bridge exists; all window.AndroidMobile.* calls found are one-directional JS→native (widget data push, reciter downloads, DND access, exact-alarm scheduling, etc.) — none wired back into any prayer-sequence function.
Conclusion: there is currently no resume/backgrounding-aware resync logic at all for the azan/iqama/black-screen/azkar chain — confirmed clean slate for your planned patch.
